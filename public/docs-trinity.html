Trinity V2: Uniswap V4 Hook Bonding Curve Protocol


Overview


Trinity is a bonding curve protocol implemented as a Uniswap V4 hook. A single hook contract serves three independent liquidity pools, each denominated in a different quote asset. The hook fully replaces the V4 AMM — it IS the liquidity source, using linear bonding curve pricing inside beforeSwap.


Because swaps route through Uniswap V4's PoolManager, Trinity pools are natively visible to the entire ecosystem: aggregators, bots, GeckoTerminal, block explorers. Trading fees are asymmetric: buy-side fees go to the multisig, sell-side fees burn $TRI.

Bonding Curve Mechanics


Curve Formula


Linear bonding curve:


price(s) = BASE_PRICE + SLOPE * s

Where s = tokens sold from that pool. Price increases linearly as supply is purchased.


• BASE_PRICE: Floor price (~$0.0001 per TRI in USD-equivalent across all pools)
• SLOPE: USDC pool has 1.5x slope (premium venue), ETH and $CHAOSLP at 1x baseline
• Buy math: Quadratic formula — solving the integral for exact token output given quote input
• Sell math: Integral from (totalSold - sellAmount) to totalSold

The steeper USDC slope makes it the "premium" venue. Arbers buy cheap on ETH/$CHAOSLP and sell expensive on USDC, generating directional fee flow and burns.


Buy / Sell


Buy: User sends quote asset (USDC, ETH, or $CHAOSLP) via the TrinityRouter. The WETH pool accepts native ETH directly — the router wraps it automatically. Price advances along the curve.


Sell: User sends $TRI via the router, receives quote asset. 1% of TRI is burned on every sell. The WETH pool can return native ETH. Price retreats along the curve.


Both directions are always available. Guaranteed liquidity at a deterministic price.


Solvency Invariant


The hook's quote reserves for each pool always equal the integral from 0 to totalSold. This holds for any sequence of buys and sells. Rounding always favors the protocol. Burns make the curve over-collateralized over time. This was verified with 27 automated tests including multi-user, cross-pool, and fuzz scenarios.



Fee Structure


1% Flat Trading Fee


Every trade across all three pools incurs a 1% fee.


On Buys: 1% in quote asset → Multisig


User buys with 1000 USDC:
  - 10 USDC (1%) --> multisig
  - 990 USDC enters the hook's reserves

All three pools send buy fees to the same multisig. The multisig then distributes: ops funding, staking gauge rewards (WETH, $CHAOSLP), etc.


On Sells: 1% in $TRI → Burned


User sells 10,000 TRI for ETH:
  - 100 TRI (1%) --> 0x000...dead (burned permanently)
  - 9,900 TRI returned to the curve
  - User receives ETH for the 9,900 TRI at curve price

Every sell is deflationary. No exceptions. All three pools burn on sells.

Trading Dynamics


Cross-Pool Arbitrage


Each pool prices TRI independently. The USDC pool's steeper slope means TRI is always most expensive there. Arbers exploit the spread:


Example (from arb simulation, tested on-chain):
  After $100K organic USDC buy:
    USDC spot: $0.003132/TRI
    WETH spot: $0.0001/TRI (unchanged)
    Spread: 3,032%

  Arb bot:
    1. Buy TRI from WETH pool for 10 WETH (~$20.6K)
       --> Gets 32.3B TRI (gentle slope = more TRI per dollar)
    2. Sell TRI to USDC pool
       --> Receives $74,817 USDC
    3. Profit: $54,197 (262% ROI)

  After arb:
    USDC spot: $0.001551 (down from $0.003132)
    WETH spot: $0.001165 (up from $0.0001)
    Spread: 3,032% --> 33% (convergence)

Every arb trade is productive:

• Burns TRI (sell side)
• Sends fees to multisig (buy side)
• Converges prices across pools

Per-Pool Sell Limits


TRI is fungible across pools, but each pool tracks its own totalSold. You can't sell more TRI to a pool than was bought from it. If an arb drains one pool's reserves, sellers exit through another pool. The hook enforces this with an ExceedsSold guard.



How Swaps Work (V4 Internals)


Pre-Settle Pattern


V4 swaps route through the PoolManager's unlock() mechanism. The TrinityRouter uses a pre-settle pattern:


1. Router settles user's input tokens to PoolManager BEFORE calling swap
2. PoolManager calls hook.beforeSwap()
3. Hook takes input from PM, computes bonding curve output, settles output to PM
4. Router takes output from PM and sends to user

For the WETH pool, native ETH is accepted: the router wraps ETH to WETH before settling. On sells, WETH is unwrapped back to ETH.

Staking


Stake TRI to earn rewards from pool fees. Uses the ChaosLPHub + RewardGauge pattern.


• Hub: Stake TRI, earn TRI (hub rewards, funded by multisig)
• WETH Gauge: Earn WETH (from ETH pool buy fees, forwarded by multisig)
• $CHAOSLP Gauge: Earn $CHAOSLP (from $CHAOSLP pool buy fees, forwarded by multisig)

Multisig collects fees from all three pools, then distributes to gauges via notifyRewardAmount(). Staking locks TRI, reducing sell pressure on curves.



Burn Mechanics


Every sell on every pool burns 1% of the TRI input. Burned tokens go to 0x000...dEaD and can never be recovered.


Burn Accounting


• Burned tokens leave circulation but totalSold only decreases by the 99% returned
• The curve retains more reserves than totalSold implies — over-collateralized
• Burns decelerate naturally as price rises (each dollar of sells burns fewer tokens)


Deployment & Governance


Multisig Control


All contracts are owned by the Rails Multisig. The multisig:


• Owns the TrinityHook (can register new pools, update fee recipients)
• Receives all buy fees across all pools
• Distributes fees to staking gauges
• Cannot withdraw bonding curve reserves (hook holds TRI, quote reserves are protocol-owned)

Summary


Trinity V2 is a Uniswap V4 hook that runs three independent bonding curves through a single contract. Swaps are visible to the entire Uniswap ecosystem — aggregators, bots, and block explorers see them natively.


• Buys: 1% fee in quote asset → multisig (WETH pool accepts native ETH)
• Sells: 1% of TRI → burned permanently
• Arb: Differentiated slopes create cross-pool arb surfaces that converge prices, burn TRI, and generate fees
• Staking: Stake TRI, earn WETH + $CHAOSLP from pool fees

The flat curve and 1% fee create a ~2% arb band across pools. Every arb trade burns supply and funds the protocol. Trading surfaces generate information corrections (arb), which generate fees, which compound as structural value.
